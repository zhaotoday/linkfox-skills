# 阶段 0：开场与边界确认

专家第一次说出需求（如"我想做一个检查 Amazon Listing 有没有侵权的 skill"），你要在 **3-5 轮内**完成开场四步：命名对齐 → 同名校验 → 情境预判 → 授权访谈。

## 第 1 步：命名对齐

### 目标

和专家商定一个符合规范又贴业务的 skill 名。

### 命名规则（agent 内部遵守）

- 必须以 `linkfox-` 开头
- 主体用小写英文 + 短横线分隔的"平台-对象-动作"结构
- 长度 ≤ 50 字符
- 贴业务：从专家一句话需求里抽关键词，不用"tool"、"utility"、"helper"这类泛词

### 命名示例

| 专家原话 | 建议名 |
|---------|-------|
| 检查 Amazon Listing 有没有侵权 | `linkfox-amazon-listing-infringement-check` |
| 帮我选能赚钱的 TikTok Shop 产品 | `linkfox-tiktokshop-opportunity-finder` |
| 看竞品这个关键词排名怎么样 | `linkfox-amazon-keyword-rank-tracker` |
| 分析评论里客户的抱怨点 | `linkfox-amazon-review-complaint-analyzer` |

### 开场话术

不要直接甩一个英文名给专家看。先用业务语言确认他要解决的是什么：

> "明白了，您是想要一个'检查 Amazon Listing 有没有侵权风险'的能力，是吗？
> 我建议给它起名叫 **linkfox-amazon-listing-infringement-check**（直译：Amazon 商品列表侵权自查）。
> 这个名字能反映您的意图吗？或者您更想叫别的？"

专家说"可以" → 进第 2 步
专家说"不合适 / 太长 / 叫简单点" → 给 2 个备选，仍不行就让专家自己取名

### 命名降级

- 专家 2 次都拒绝你的提议 → 直接请专家自己取（中文也可，你之后转成英文命名）
- 专家完全定不下来 → 用草稿名 `linkfox-draft-YYYYMMDD-NNNN`，告诉专家"先用个临时名，做完再改"

## 第 2 步：同名校验

### 目标

防止覆盖已有 skill 导致专家之前的经验丢失。

### 操作

用文件系统工具检查 `skills/<name>/SKILL.md`（在合适的 skills 目录下）是否存在：

- 路径不存在 → 直接往下走
- 路径存在 → **必须停下来问专家**：

```
我注意到已经有一个同名的 skill：
  名称：linkfox-amazon-listing-infringement-check
  描述：<读出已有 SKILL.md 的 description 前 100 字>
  创建时间：<文件 mtime>

您希望：
  ① 覆盖这个老的，从头做一个新的（老内容会丢失）
  ② 保留老的，把新的叫一个不一样的名字
  ③ 先不做，我去看看老的是什么
```

### 同名不止一次

如果相似前缀有多个（如 `linkfox-amazon-listing-check`、`linkfox-amazon-listing-audit`），列出来让专家确认是不是重复建设，再决定继续或合并。

## 第 3 步：情境预判

### 目标

在正式开问之前，让专家"对齐焦点"，避免后面问 D1 时走到岔路。

### 方法

用一两句话复述你对这个需求的理解：

> "我理解这个 skill 要做的事情是：当您拿到一个 Amazon 商品链接，您想让它自动帮您判断这个商品有没有品牌、描述、或协议层面的侵权风险，然后给您一份清单。
> 这个理解对吗？有没有我猜歪的地方？"

专家点头 → 进第 4 步
专家纠正 → 吸收纠正，重新说一次 → 再确认

### 边界：别过度预判

- 不要在第 3 步就开始问"您平时怎么判断品牌字段"——那是 D3 的事
- 只确认"大方向对不对"，不挖细节

## 第 4 步：授权访谈

### 目标

让专家知道接下来要做什么、耗时多久、能不能暂停。

### 话术

> "接下来我会问您 5 个方面的问题，大概 30-45 分钟：
>   ① 这个能力用在什么场景下
>   ② 您每次用到它时，手头有什么资料
>   ③ 从看到资料到做出判断，您脑子里走过哪几步
>   ④ 哪些规则是硬性的，哪些是经验感觉
>   ⑤ 什么情况下这个能力不该用
>
> 中间您随时可以说'不想答这个''我没时间了'，我会保存进度，下次继续。
> 咱们开始吗？"

专家说"好 / 可以 / 开始吧" → 写入进度标记 → 进入阶段 1

## 开场期的硬约束

- 不在此阶段问任何 D1-D5 的具体内容
- 不谈 skill 的文件结构、YAML、脚本这些词
- 不承诺具体耗时之外的东西（不要说"我会给您生成一份精美的..."）

## 骨架产物（阶段 0 结束时）

```
skills/<name>/
└── references/
    └── interview-record.md   ← 只含 frontmatter + 进度标记
```

interview-record.md 的初始内容：

```yaml
---
schema_version: 1
skill_name: <confirmed-name>
expert: <expert-identifier-if-known-else-unknown>
```

## 典型卡点与处理

| 卡点 | 处理 |
|-----|------|
| 专家直接开始讲业务细节，跳过命名 | 打断："等等，我们先给这个能力起个名字，否则文件存哪都不知道" |
| 专家问"这个 skill 会同步到别人那里吗 / 怎么分发" | 回答："这个 skill 会先存在您本地，分发的事情后面另外安排，跟我们现在的访谈没关系" |
| 专家问"要不要现在就演示一下" | 回答："访谈完我会帮您做出来，您到时候试用就行。现在先把方法说清楚，之后改起来才不会乱" |
| 专家没授权就开始抱怨一堆细节 | 记下来但不结构化，等他说完再走第 4 步正式授权 |
