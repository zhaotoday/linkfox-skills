# 阶段 1：访谈（5 维必挖）

访谈的核心不是"问完一张清单"，而是"把专家脑子里的隐性判断挖到能落地"。5 个必挖维度 **D1-D5 缺一不可**；每维内部你自由选择问法、顺序和深度。

## 全局原则（每轮都要记住）

1. **骨架是维度，不是问题列表**。你可以用 10 种问法挖 D1，只要挖到判定标准
2. **触发式复述，不是机械复述**。只在专家说了关键判断时复述一次确认，不要每轮都复述
3. **遇到模糊就追问**。专家说"差不多"、"一般来说"、"有时候"，你必须追到具体
4. **规则后反向追问**。专家说一条规则，跟一句"您见过反例吗？"
5. **原话保留到 `expert_verbatim`**。**禁止润色改写**，包括口头语、方言、省略句
6. **自己补的标 assumed**。你根据上下文补的字段，列进 `assumed_fields`
7. **每维完成写进度**。挖够一维，更新 `<!-- interview-progress -->` 标记

## 深挖四式（随时调用）

### 1. 复述校验（触发式）

专家说完一个关键判断后，你复述一次：

> 专家："这时候我会看品牌字段，如果是 generic brand 就要小心。"
> 你：  "我理解您是说——当商品的 brand 字段写着 'generic brand' 时，需要额外警觉，对吗？"

触发条件：
- 专家讲了一条决策规则
- 专家给了一个判断的"阈值"（"低于 4 星"、"超过 30%"）
- 专家说了反直觉的内容（"一般人以为 X，但实际是 Y"）

不触发的情形：
- 专家在讲背景信息（"我做这行 5 年了"）
- 专家说"嗯对"、"是"等连接句

### 2. 歧义追问

专家用了模糊词，马上追：

| 模糊词 | 追问模板 |
|-------|---------|
| "差不多" | "差不多的边界大概是多少？比方说 10% 算，20% 算吗？" |
| "一般来说" | "不一般的情况通常是什么样的？" |
| "有时候" | "有时候大概是多大比例？您上次碰到是什么场景？" |
| "这个要注意" | "具体是哪一步容易踩？" |
| "经验判断" | "能不能举一个最近的例子？您当时看到什么决定这么判的？" |
| "看情况" | "哪几种情况会让您选 A，哪几种选 B？" |

### 3. 反向追问

专家说规则时，跟一句反例：

> 专家："标题里必须含核心关键词，否则没流量。"
> 你：  "您有没有见过不含核心关键词，但流量反而还行的 listing？如果有，那种一般是什么情况？"

作用：
- 把硬规则和经验规则分离（硬规则专家通常说不出反例，经验规则一定有反例）
- 暴露边界条件（D5 的素材往往从反例里浮出来）

### 4. 原话保留

当专家说出有价值的核心判断，**原封不动**写入 `expert_verbatim`：

```yaml
- step: 3
  name: 品牌关键词扫描
  expert_verbatim: |
    这一步我主要看两个东西，一是 title 里头有没有竞品的品牌词，
    比方说 anker、baseus 这种明确商标；二是描述里有没有暗示
    "适用于某品牌"这种措辞，这种不是直接侵权但可能被投诉。
```

不允许的"改写"：
- 去掉口语（"这一步我主要看..." → "本步骤检查..."）
- 归纳改写（"一是...二是..." → "检查标题与描述中的品牌词及暗示性措辞"）
- 删减细节（去掉"比方说 anker、baseus"这种具体举例）

## 必挖 5 维的判定标准

### D1 业务情境

**目标**：回答"这个 skill 在什么真实情境下被调用"。

**挖够的标志**：能用一句话写出"在 **X 情境** 下，针对 **Y 输入**，专家希望得到 **Z 结果**"。

**引导问法举例**（选 1-2 个用就行）：
- "您平时什么时候会想起来用这个能力？"
- "最近一次您做这件事，是因为遇到了什么？"
- "假设一天里有这么一瞬间您特别想要一个工具帮您做这个——那一瞬间是什么时刻？"

**需要追问**：
- 专家说"选品的时候" → "选品里的具体哪一步？看竞品时、看自己 listing 时、上新前、上新后？"
- 专家说"平常都用" → "上周您实际用了几次？最后一次是做什么事？"

**达标示例**：
> scenario: 在上架新品前做最终合规检查，以免 listing 上线后被投诉侵权下架。
> primary_user: Amazon 运营
> typical_trigger_moment: 完成 listing 初稿，进入"提交审核"前最后一步
> success_criteria:
>   - 发现明显的品牌侵权点
>   - 发现描述中可能被投诉的措辞
>   - 给出修改建议

### D2 输入形态

**目标**：列出专家每次用这个能力时手头的典型资料。

**挖够的标志**：至少 1 种典型输入，每种有样例或具体描述。**越具体越好**。

**引导问法举例**：
- "您用这个能力的时候，手头一般有什么？"
- "是一个链接？一个 ASIN？一份 Excel？还是一张截图？"
- "能给我举一个最近的例子——上次您做这件事时粘贴或打开的是什么？"

**需要追问**：
- 专家说"就是个链接" → "链接是 Amazon 自己的，还是从 Google 搜的、还是从聊天群里复制的？"
- 专家说"Excel 表" → "表里有哪几列？能描述一下格式吗？"

**达标示例**：
> inputs:
>   - type: url
>     example: https://www.amazon.com/dp/B0XXXXXX
>     notes: 必须是商品详情页链接，不是搜索结果页
>   - type: asin
>     example: B0XXXXXX
>     notes: 当有 ASIN 但没链接时

### D3 核心流程

**目标**：列出从输入到结论的可命名步骤；每步判定 `execution_type`（三选一：script / llm / delegate）。

**挖够的标志**：
- 至少 3 个步骤
- 每步有名称 / 输入 / 决策 / 输出
- **每步标 `execution_type: script | llm | delegate`**（严格三选一）
- `script` 步骤有明确的输入 → 输出规则
- `delegate` 步骤有 `delegates_to` + 专家确认原话

**引导问法举例**：
- "从您拿到这个链接，到您做出'可以上架'或'不能上架'的判断，中间经过几步？"
- "您脑子里第一件事做什么？然后呢？"
- "您会不会跳过某些步骤？什么情况下会跳？"

**关键追问（每步都要走一遍三分法）**：

**第一问——这步是不是要用到现成的专业工具？**

触发条件详见 `06-delegate-discovery.md`（提到平台名、查/筛/采/评/对比 + 领域对象、"我一般用 XX 工具看……"等）。

触发 → **你必须立即去搜**：

```bash
linkfoxskill search <关键词>
```

命中候选 → 用业务语言读给专家（**不读 slug**）："这一步我找到 N 个现成能做的小助手：1. ……。您平时用吗？哪个合您意？"

- 专家明确采纳 → 标 `execution_type: delegate`，填 `delegates_to` + `delegate_inputs_mapping` + `delegate_confirmed_at` + `delegate_expert_quote`，同时把 slug 加进顶层 `depends_on`
- 专家拒绝 / 没用过 / 说"再看看" → 落空，进入第二问
- 电商库无果 → 退到 `/find-skills` 通用搜索 → 仍无果 → 询问专家三选一（程序 / 临场判断 / 换工具名再搜）

无论命中与否，每次搜索都追加一条到顶层 `delegate_search_log`。

**第二问（第一问走 delegate 则跳过）——固定规则还是经验感觉？**

> "这一步您是按一个固定规则来判断，还是靠经验感觉？
> 比方说'标题超过 200 字'就是固定规则；
> '这个标题读起来像垃圾'就是经验感觉——您这步是哪种？"

- **固定规则** → 标 `execution_type: script`，**这一步会生成 Python 脚本**
- **经验感觉** → 标 `execution_type: llm`，留给下游 agent 的语言理解处理
- **专家拿不准** → 你主动给示例："像'检查价格是不是超过 999 美元'这种算固定；
  '这个价格合不合理'就算经验——您这步更像哪一种？"

**委托候选的搜索、展示、专家确认、采纳落库，详见 `references/06-delegate-discovery.md`**。

**需要追问**：
- 专家说"看一下" → "看什么字段？看到什么算过，看到什么要停下来再查？"
- 专家说"然后就判断一下" → "您怎么判断的？凭什么做这个决定？"

**达标示例**：
> process:
>   - step: 1
>     name: 解析商品页提取字段
>     inputs: [product_url]
>     decisions: [从 HTML 中抽取 title, brand, description, bullet_points]
>     outputs: [structured_fields]
>     execution_type: script
>     script: scripts/parse-amazon-listing.py
>     expert_verbatim: |
>       我就是进到商品页，把标题、品牌、描述这几个地方复制出来
>     assumed_fields: []
>   - step: 2
>     name: 拉取竞品同款商品数据
>     inputs: [keyword, marketplace]
>     decisions: [按月销 ≥ 500 筛选，取前 100 个]
>     outputs: [competitor_list]
>     execution_type: delegate
>     delegates_to: linkfox-sellersprite-product-search
>     delegate_inputs_mapping:
>       keyword: keyword
>       marketplace: marketplace
>       minUnits: 500
>       size: 100
>     delegate_confirmed_at: 2026-04-27T10:42:15+08:00
>     delegate_expert_quote: |
>       就用卖家精灵那个，我平时都用它看竞品
>     expert_verbatim: |
>       第二步按某些条件查 100 个竞品，销量要 500 以上
>   - step: 3
>     name: 描述措辞风险判断
>     inputs: [description, bullet_points]
>     decisions: [判断是否有"适用于 XX 品牌"、"等同于 XX"等暗示性措辞]
>     outputs: [description_risk_notes]
>     execution_type: llm
>     expert_verbatim: |
>       描述里不能出现'适用于 apple'这种，但'适用于多种设备'就可以，这个要看上下文

### D4 关键规则

**目标**：列出至少 3 条硬性规则或经验规则，每条带反例或适用场景。

**挖够的标志**：
- ≥ 3 条规则
- 每条标 `hard` 或 `soft`
- 每条有 `counter_example` 或 `scope`

**引导问法举例**：
- "您刚才讲流程的时候提到几条规则，我梳理一下——[复述]——还有其他规则吗？"
- "有没有什么事情是您宁可少做一步也绝不会违反的？那是硬规则。"
- "有没有什么是'通常这样，但偶尔例外'的？那是软规则。"

**必触发反向追问**：
> "您能想起一次规则 X 的反例吗？什么情况下您会违背这条规则？"

- 专家说"没有反例，这就是硬规则" → `type: hard`
- 专家说"嗯……有时候 X 情况下我会放宽" → `type: soft` 且 `counter_example` 填反例

### D5 边界与踩坑

**目标**：至少 2 个不适用场景 + 至少 1 个踩坑案例。

**挖够的标志**：
- ≥ 2 条 `boundaries`（不适用场景）
- ≥ 1 条 `pitfalls`（反直觉踩坑）

**引导问法举例**：
- "什么情况下这个能力**不应该**用？"
- "您见过新人用这个方法翻车的例子吗？"
- "如果我拿这个能力去做 [延伸场景]，会不会出问题？"

**必触发的追问**：
- 专家只给 1 个不适用场景 → "还有别的情况吗？比如输入资料不完整的时候、比如涉及多语言的时候？"
- 专家说"没踩过坑" → "那您新人时期最容易犯的错是什么？"

**达标示例**：
> boundaries:
>   - scenario: 商品链接打不开 / 已下架
>     why_not_applicable: 无法提取字段，没有判断依据
>     expert_verbatim: |
>       链接都挂了你让我查个啥
>   - scenario: 多语言描述（日语、法语等非英文）
>     why_not_applicable: 规则基于英文品牌词和措辞，其他语言可能漏判
>     expert_verbatim: |
>       日语的我真的看不懂，我一般直接让别的同事看
>
> pitfalls:
>   - scenario: brand 字段写 unbranded 就以为安全
>     why_wrong: unbranded 不是免死金牌，如果描述里还是抄了别人的措辞一样会出事
>     expert_verbatim: |
>       很多新人以为 brand 写 unbranded 就没事，其实 listing 整体还是得看

## 可选 2 维

### D6 输出形态偏好

**触发条件**：专家明确提到"我想要表格"、"给我个清单"、"写成报告"等。

**不触发时**：默认用"结构化清单"格式，不主动追问。

### D7 外部依赖（仅用于"自己封装 API"场景）

**触发条件**：D3 中某步 `execution_type=script` 且需要调外部平台接口、抓网页、查数据库等。

**优先级**：能走 delegate 就走 delegate（见 D3 第一问和 `06-delegate-discovery.md`）；只有现成 skill 不存在 / 专家拒绝 / 搜索兜底链全空时，才退到这里自己封装。

**不触发时**：默认 skill 只用本地数据 + LLM 推理。

## 原话片段的归档位置

每轮访谈的完整对话（你 + 专家）按时间顺序写入 interview-record.md 的正文区（frontmatter 后面）。结构化字段在 frontmatter，原稿在 body。

```markdown
---
schema_version: 1
... (结构化字段)
---

## 第 1 轮访谈（2026-04-27 10:30）

**你**：您平时什么时候会用这个能力？
**专家**：一般是上架前最后一步，我会让实习生跑一遍...
...
```

## 降级策略

| 情况 | 处理 |
|------|------|
| 专家每维只给一句话、没干货 | 每维记下这一句就进下一维；阶段 3 生成时在 frontmatter 标 `quality: partial` |
| 专家拒答某维（"我不想说规则"） | 该维标 `refused`，继续；frontmatter 标 `quality: partial` |
| 专家中途说"没时间了" | 写进度标记，礼貌结束；下次可续 |
| 专家越答越烦躁（短句越来越多） | 降温：减少复述，每维问 1-2 句就走；每维满足最低 1 条原话即可 |
| 专家给的全是大话（"就是要选好的产品"） | 反复要具体例子："您最近一次做，看的是哪个 ASIN？" |

## 硬禁忌

- 不在访谈中讨论 skill 的文件结构、YAML 语法、脚本参数等
- 不自己"演绎"专家的原话（例如把"他们都这么搞"改写成"行业普遍实践"）
- 不在一维没挖到判定标准时就跳到下一维（除非被 refused / 降级）
- 不承诺不知道的事（"我帮您对接平台接口"——除非 D7 明确确认过）
