# 阶段 2：结构化 + 假设批量确认

阶段 1 结束时，你手里已有一份半结构化的 `interview-record.md`：5 维都有内容，但其中一部分字段是**你自己补的**，一部分可能**含敏感信息**，另外 5 维的整体叙事**专家还没校对过**。

阶段 2 按**固定顺序**做三件事：**隐私审查 → 批量假设确认 → 最终校对**。三件事都通过，才进阶段 3。

## 第 1 件事：隐私审查

### 目标

不把专家或公司的敏感信息留在 skill 里，因为 skill 目录会被同步到云端、被别人看到。

### 话术

> "在我把这些内容整理成正式文件前，想跟您确认一下：
> 刚才您提到的内容里，有没有**不方便让别人看到**的部分？
> 比方说：
>   · 您公司的内部操作 SOP、独家方法论
>   · 具体的竞品名字、供应商名字、员工姓名
>   · 您自己账号的一些数据
>   · 涉及商业谈判、客户抱怨的具体案例
>
> 如果有，我帮您在最后的文件里打码或者改成泛指。"

### 专家的三种可能回答与处理

**A. 专家说"没有，都可以"**
→ 写一行备注到 interview-record：`privacy_review: no_sensitive_declared_at_<timestamp>`
→ 进第 2 件事

**B. 专家指出某几处**
→ 对应字段的 `expert_verbatim` 做**最小范围**处理：
  - 具体公司名/人名 → 打码为 `[某品牌]` / `[某同事]`
  - 具体数值如果是公司独家（如毛利率） → 改为量级描述（"20%-40%"）
  - 整段内部流程（如"我们公司的 QA 流程"） → 征求意见："这段我整个删掉还是改成通用描述？"
→ 每处处理完，在原字段下加 `redacted: true` 并记改动原因

**C. 专家不确定**
→ 逐条读 D1-D5 的 `expert_verbatim`，每一段问："这段可以公开吗？"
→ 耐心过一遍，宁可慢也不要漏

### 硬约束

- **不允许你自行决定什么算敏感**。只问专家，由专家判断
- **打码必须可逆**：interview-record.md 保留原文在另一字段 `verbatim_original`（本地不上传时可保留），公开副本用 `expert_verbatim`；若打码即删除，写 `redaction_note` 说明
- **发现明显敏感但专家没指出的**（比如专家报了邮箱、手机号），主动提醒："您刚才提了一个邮箱地址，要不要也打码？"

## 第 2 件事：批量假设确认

### 目标

让专家一次性过完你补齐的所有字段，避免后面发现误解。

### 准备工作

把 interview-record.md 中所有 `assumed_fields` 汇总成一张清单：

```
我在整理过程中，根据上下文替您补了以下字段。
请过目一下，有不对的告诉我：

D2（输入形态）第 1 条
  您说过"就是个链接"，我补了：
    · type: url
    · example: https://www.amazon.com/dp/B0XXXXXX
    · notes: 必须是商品详情页链接
  这个理解对吗？ [ ]对 [ ]不对

D3（流程）步骤 2
  您说"我看品牌字段"，我补了：
    · decisions: [如果 brand 含"generic"则标红；如果 brand 是第三方商标则标红]
  这个对吗？ [ ]对 [ ]不对

D3（流程）步骤 2 脚本标注
  根据您说"brand 如果写 generic brand 那肯定不行"，我判定这步是固定规则（能用脚本实现）。
  对吗？ [ ]对（生成脚本） [ ]不对（改为经验判断）

... （逐条列出）
```

### 展示原则

- **按维度分组**，不按字段类型分
- **每条配原文**：显示你补的字段 + 专家对应的 `expert_verbatim`
- **单选项清晰**：只问"对 / 不对"，不开放式问

### 专家的反应

**A. 全部"对"**
→ 在 interview-record 加 `<!-- assumptions_confirmed: round=N all=true -->`
→ 进第 3 件事

**B. 指出几条不对**
→ 针对每条不对的：
  - 问"正确的应该是什么？"
  - 吸收修改，更新对应字段，从 `assumed_fields` 中移除
  - 若涉及 `deterministic` 判定变化（本来标 true，专家说"其实是经验判断"）：
    - 删除对应 `script` 字段
    - 改为 `deterministic: false`
→ 修改完再过一遍新列表，直到专家全部确认

**C. 专家说"差不多都对，别问这么细"**
→ 温和坚持："这些假设如果不确认，后面生成出来的东西可能跟您想的不一样。
  我保证过得快——只挑您自己没明说的那几条确认，5-8 条就完"
→ 实在不配合 → 只确认 D3 的 `deterministic` 判定（这个最关键），其他标 `assumptions_skipped_by_user`

## 第 3 件事：最终校对

### 目标

专家对 5 维的整体叙事点头，你才能进阶段 3 生成文件。

### 展示方式

用**业务语言**（不含任何技术术语），给出 5 维概览：

```
好的，我把您的经验整理成了这样。请您最后过一眼，有不对的说一声：

🎯 用途（D1）
  这个能力帮您在上架新品前做最后的合规自查，
  输入是一个 Amazon 商品链接，输出是一份"可能有侵权风险的点"清单。

📥 您的资料（D2）
  ① 商品详情页链接（主要）
  ② 或者纯 ASIN 编号（备用）

🔧 操作步骤（D3）—— 共 3 步
  ① 从页面抽出标题、品牌、描述——这一步按固定规则，我会用脚本实现
  ② 检查品牌字段是否 generic 或第三方商标——固定规则，脚本实现
  ③ 判断描述措辞是否有暗示性侵权——需要上下文理解，由我来判断

📏 规则（D4）—— 共 4 条
  硬规则：
    · 品牌字段含 "generic" 直接标红
    · 描述出现特定商标名且非授权使用直接标红
  软规则：
    · 商品类别是配件类时，品牌审核可以放宽
    · 促销词密度过高（>3 个）时倾向标黄

⛔ 不适用（D5）
  · 商品链接打不开 / 已下架
  · 多语言非英文描述

⚠️ 踩坑
  · unbranded 不是免死金牌，描述措辞也要一起看

这个整体对吗？有没有遗漏或者需要改的？
```

### 展示的注意事项

- 用 emoji / 符号分块视觉引导（上面 🎯📥 这种），但**只在校对展示时用**，不进 skill 文件
- 不用中括号列字段名（"D1: scenario:"这种），完全业务语言
- 显式区分硬规则、软规则
- 标出"我会做的（脚本）"和"我判断（LLM）"两类步骤

### 专家的反应

**A. "可以 / 没问题 / 对"**
→ 在 interview-record 写 `<!-- final_review_passed: <timestamp> -->`
→ 写 `<!-- interview-progress: round=N dim_done=[D1,D2,D3,D4,D5] status=ready_to_generate -->`
→ 进阶段 3

**B. 指出某处要改**
→ 精确定位到哪一维哪一条 → 修改对应字段 → 重新展示**改动后的那一块**确认 → 通过后再进阶段 3
→ 不要整个 5 维重念一遍（耐心损耗）

**C. 专家说"大致可以，你看着办吧"**
→ 不接受。坚持："大致可以的意思是有 80% 对，那 20% 我很想知道是哪里。您哪怕挑一个您最不放心的地方告诉我"
→ 若专家真的完全不想校对 → 记 `final_review_skipped_by_user: true`，在 skill frontmatter 打 `quality: partial` 提醒风险

## 进入阶段 3 的前置检查（自检清单）

全部满足才进阶段 3：

- [ ] `privacy_review` 标记已写
- [ ] 所有 `assumed_fields` 要么已确认，要么被移除，要么标 `assumptions_skipped_by_user`
- [ ] `final_review_passed` 或 `final_review_skipped_by_user` 二选一已写
- [ ] D3 每个步骤的 `deterministic` 字段已定稿
- [ ] 进度标记更新为 `status=ready_to_generate`

## 硬禁忌

- **不跳过隐私审查**。哪怕专家说"都可以"，也要显式问一次
- **不把"我补了"藏起来**。所有 assumed 字段必须显式列给专家看
- **不用技术术语展示校对**。专家看到的是业务叙事，不是 YAML
- **不在阶段 2 补新字段**。阶段 2 只确认、不新增（专家主动提新内容除外，此时记录并显式问"这条要加进去吗？"）
