# 阶段 4：交接（触发评估器）

skill 文件已经全部落盘、自检通过。最后一步是告诉专家"做完了"，并询问是否评估。

## 交接话术模板

### 主干话术

> "**linkfox-<name>** 已经创建好了，位置在本地 skills 目录下。
> 它包含以下东西：
>
> · 一份使用说明（SKILL.md）
> · 一份完整的操作流程（references/workflow.md）
> · 一份规则与边界清单（references/rules-and-boundaries.md）
> · 您的原始访谈记录（references/interview-record.md）
> · <若有脚本>  <N> 个用来保证结果稳定的小程序（scripts/<list>）
>
> 现在要不要我帮您**评估一下质量**？
> 评估会模拟一次真实使用，检查这个 skill 是不是真按您说的那样工作，大概几分钟。"

### 展示时的注意

- **不暴露技术术语**。"YAML frontmatter"、"structured fields" 这种别出现
- **脚本要说人话**。"Python 脚本" → "小程序 / 自动检查" / "用来保证结果稳定的一段代码"
- **给选择权**。"要不要评估"是可选的，不强推

## 专家的 3 种回答与处理

### A. 同意评估

> "好，那我帮您评估。"

**行动**：
1. 用自然语言唤起 `linkfox-skill-evaluator`，例如：
   > "（向另一个 agent 会话）请对 `linkfox-<target-name>` 这个 skill 做评估，报告保存到 `skills/<target-name>/.eval/`。"
2. 或直接执行评估器的 SKILL.md 入口（若评估器已加载）
3. 告诉专家："评估已开始，大概几分钟出报告。"

**触发评估器的触发语建议**（给 agent 的指令层面）：
- "对 linkfox-<name> 做一次完整评估"
- "评估一下 linkfox-<name> 这个 skill"
- "跑一遍 linkfox-<name> 的四维评估"

### B. 拒绝评估

> "先不用了 / 我先自己看看 / 过几天再评估。"

**行动**：
1. 告诉专家怎么之后自己触发：
   > "好的。等您想评估时，可以对我说'帮我评估一下 linkfox-<name>'，
   > 或者让另一个 agent 会话来做这件事。"
2. 收尾："需要改什么随时叫我。"
3. 结束会话

### C. 专家说"我想先试用一下"

> "我想先用一下看看效果。"

**行动**：
1. 礼貌说："当然可以。"
2. 若专家有具体的测试输入，引导他试用这个 skill：告知他怎么在他的 agent 环境下触发（"您下次跟 agent 说'帮我查侵权：<链接>'就会自动走这个 skill"）
3. 提醒："试用后如果发现不对的地方，可以再叫我调整，或者跑一次评估让我知道问题在哪"
4. 结束会话

## 评估完之后的闭环（若专家选了评估）

评估器运行完，会在 `skills/<target-name>/.eval/report-<timestamp>.md` 产出一份报告。报告的 `publish_advice` 字段有三种：

| publish_advice | 含义 | 你（Creator）下一步 |
|---------------|------|-------------------|
| `ok` | 质量合格，可发布 | 告诉专家："评估通过了，没问题。" 结束 |
| `revise` | 质量基本可以，但有改进建议 | 展示报告的 top 3 issue，问："要不要现在跟我一起改改这几个点？" |
| `block` | 有严重问题，不建议直接用 | 明确告诉专家："评估发现几个比较严重的点，建议改完再用：[列出 issue]。要不要我们一起改？" |

### 进入迭代模式（`revise` 或 `block` 且专家同意修改）

1. 读报告中具体的 issue 列表，映射回 Creator 的阶段：
   - E1 忠实度低（skill 内容与 interview-record 有出入） → 回到阶段 2 对照修正
   - E2 可读性低（SKILL.md 过长 / 结构乱） → 回到阶段 3 重新生成，精简展开
   - E3 契约问题（脚本缺失、字段缺失等） → 回到阶段 3 按 lint 报告修
   - E4 行为问题（规则不足、边界漏洞） → 回到阶段 2，针对性补问专家
2. 改完重新触发评估；如此循环直到 `ok`

### 迭代上限

- 默认最多 3 轮迭代
- 3 轮后若仍 `revise`，告诉专家："我已经改了 3 轮，剩下的这些可能需要您再补充一些具体经验才能解决。要不要先停一停，回头再来？"
- 3 轮后若仍 `block`，升级："这个 skill 的基础信息可能不够做成可靠版本，建议我们回到阶段 1 再补挖几维。您愿意吗？"

## 收尾的硬约束

- **不自吹**。不要说"我给您生成了一份精美的 / 完美的 skill"；只陈述事实
- **不遗漏脚本**。如果 skill 有 scripts/，必须在交接时口头提到（让专家知道有"自动化检查"保底）
- **不预设专家一定评估**。平等给出"评估 / 不评估 / 先试用"三个选项

## 交接完成的痕迹

在 interview-record.md 末尾写：

```
<!-- interview-progress: round=N dim_done=[D1,D2,D3,D4,D5]
     structure_confirmed=true
     generated=true
     handoff_done=true
     handoff_choice=evaluate|skip|tryout
     timestamp=<ISO-8601>
     status=complete -->
```

至此本次 Creator 会话结束。
